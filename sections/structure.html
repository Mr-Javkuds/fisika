Mengapa Code Structure Penting?

üìö Apa itu Code Structure?

Code Structure adalah cara kita mengorganisir dan memformat code agar readable, maintainable, dan consistent. Ini bukan hanya soal "looks pretty", tapi fundamental untuk professional development.

Think of it as "grammar and punctuation" untuk programming. Bad structure = working code tapi nightmare untuk maintain!

‚úÖ Mengapa Structure Matters? (CRITICAL!)

1. Code is Read 10x More Than Written

You'll spend WAY more time reading code than writing it. Good structure = faster comprehension.

2. Team Collaboration

Consistent structure means everyone can understand each other's code instantly.

3. Bug Prevention

Well-structured code makes bugs obvious. Messy code hides bugs in complexity.

4. Professional Standard

Following conventions (like PEP 8) shows you're professional, not amateur.

üìñ PEP 8: Python's Style Guide

PEP 8 adalah official style guide untuk Python. Bukan "suggestions" - ini adalah industry standard yang diexpect oleh setiap Python developer.

Naming Conventions
  • ‚Ä¢ Variables: snake_case
  • ‚Ä¢ Functions: snake_case
  • ‚Ä¢ Classes: PascalCase
  • ‚Ä¢ Constants: UPPER_CASE
Spacing Rules
  • ‚Ä¢ Indentation: 4 spaces
  • ‚Ä¢ Max line: 79 characters
  • ‚Ä¢ Blank lines: 2 before class, 1 before method
Import Order
  • 1. Standard library
  • 2. Third-party
  • 3. Local imports
  • ‚Ä¢ Alphabetical within groups

üéØ The "Why" Behind Structure Rules

Why snake_case for variables?

More readable than camelCase in Python. "velocity_x" clearer than "velocityX".

Why 4-space indentation?

2 spaces = hard to see nesting. 8 spaces = too much horizontal space. 4 = perfect balance.

Why import order matters?

Instantly see dependencies. Standard lib (always available) vs third-party (need install) vs local (your code).

Why max 79 characters per line?

Fits on any screen, readable without horizontal scrolling, side-by-side code review.

‚ö° Key Principle: "Code for Humans, Not Machines"

Computers don't care about formatting. Humans do! Good structure makes code self-documenting. If you have to choose between "clever" vs "clear", ALWAYS choose clear.

  • ‚Ä¢ Group: standard ‚Üí third-party ‚Üí local

  • ‚Ä¢ Alphabetical order dalam each group
  • ‚Ä¢ Avoid wildcard imports (from x import *)
