Tipe Data Python

Dari basic types hingga advanced collections - semua dengan contoh real dari kode momentum 2D

Apa itu Tipe Data?

üìö Definisi

Tipe Data (Data Types) adalah classification yang menentukan jenis nilai yang bisa disimpan dalam variable dan operasi apa yang bisa dilakukan pada nilai tersebut. Think of it as "aturan main" untuk setiap variable - apakah itu number, text, true/false, atau collection of items.

‚úÖ Mengapa Tipe Data Penting?

1. Type Safety

Mencegah operasi invalid: "5" + 3 adalah error karena string + int tidak kompatibel.

2. Memory Efficiency

Different types use different memory. Integer lebih efficient dari string untuk numbers.

3. Code Clarity

Type hints make code self-documenting: mass: float jelas lebih informative.

4. Performance

Right type = faster operations. NumPy arrays jauh lebih cepat untuk math operations vs lists.

üéØ Python's Type System

Dynamic Typing

Variables can change type:

x = 5
x = "hello" # OK!
Strong Typing

No implicit conversions:

"5" + 3 # ERROR
int("5") + 3 # OK!
Duck Typing

"If it quacks like a duck..."

Any object with __iter__
can be looped

üèóÔ∏è Data Type Categories

Primitive Types
  • ‚Ä¢ int - Whole numbers: 42, -17
  • ‚Ä¢ float - Decimals: 3.14, 0.01
  • ‚Ä¢ str - Text: "Hello", 'Python'
  • ‚Ä¢ bool - True/False: True, False
Collection Types
  • ‚Ä¢ list - Ordered, mutable: [1, 2, 3]
  • ‚Ä¢ tuple - Ordered, immutable: (1, 2)
  • ‚Ä¢ dict - Key-value pairs: {"x": 5}
  • ‚Ä¢ set - Unique values: {1, 2, 3}

Functions & Methods

Function definition, type hints, docstrings, callbacks - Production-ready patterns

Apa itu Functions?

Functions (fungsi) adalah reusable blocks of code yang perform specific task. Think of them as mini-programs dalam program. Kamu write code sekali dalam function, lalu call function itu berkali-kali whenever needed.

‚úÖ Why Use Functions?

  • ‚Ä¢ DRY Principle: Don't Repeat Yourself - write once, use many times
  • ‚Ä¢ Modularity: Break complex problems into smaller pieces
  • ‚Ä¢ Readability: calculate_momentum() lebih jelas dari inline code
  • ‚Ä¢ Testing: Easier to test individual functions
  • ‚Ä¢ Reusability: Use same function in different parts

üéØ When to Create Function?

  • ‚Ä¢ Repetition: Copy-paste code 3+ times? Make it a function!
  • ‚Ä¢ Complexity: Code block doing complex task? Extract to function
  • ‚Ä¢ Logical Unit: Self-contained task? Perfect for function
  • ‚Ä¢ Readability: Name describes what code does better than code itself

üìê Function Anatomy

def function_name(parameter: type) -> return_type:
    """Docstring: what function does"""
    # Function body
    result = do_something(parameter)
    return result  # Optional

# Call the function
output = function_name(input_value)

Components: def keyword, name, parameters, type hints, docstring, body, and optional return value.

Python Fundamentals #3

Loops & Control Flow

For, while, enumerate, comprehensions, try-except - Flow control mastery

Apa itu Loops & Control Flow?

üìö Definisi

Loops adalah struktur yang mengulang block of code berkali-kali. Control Flow adalah mechanism yang menentukan urutan eksekusi code - which code runs, when it runs, and how many times it runs.

Think of it as "program's decision-making brain" - loops handle repetition, conditionals handle choices, and together they control program flow.

‚úÖ Mengapa Loops & Control Flow Penting?

1. DRY Principle

Instead of copy-paste code 100x, loop once! Saves lines and reduces errors.

2. Dynamic Behavior

Handle unknown quantities. Process 10 items atau 10,000 items dengan same code.

3. Decision Making

Programs must make choices: if collision, reverse velocity. Control flow enables this.

4. Real-time Systems

Event loops keep apps responsive, update UI every frame, process user input continuously.

üîÑ Loop Types in Python

For Loop

Iterate over sequences (known count)

for item in collection:
    process(item)

Use when: You know what to iterate over

While Loop

Repeat until condition false (unknown count)

while condition:
    do_something()

Use when: You don't know iterations count

üéØ Control Flow Structures

Conditionals (if-elif-else)

Make decisions based on conditions

if ball_colliding:
    reverse_velocity()
elif out_of_bounds:
    reset_position()
else:
    continue_moving()
Exception Handling (try-except)

Handle errors gracefully without crashing

try:
    value = float(input)
except ValueError:
    show_error("Invalid input!")

üí° Best Practices

‚úÖ DO
  • ‚Ä¢ Use for for iterables
  • ‚Ä¢ Use enumerate() when need index
  • ‚Ä¢ Guard clauses (early return) to reduce nesting
  • ‚Ä¢ Break complex loops into functions
‚ùå DON'T
  • ‚Ä¢ Don't modify list while iterating it
  • ‚Ä¢ Avoid infinite loops without break condition
  • ‚Ä¢ Don't nest loops >3 levels deep
  • ‚Ä¢ Don't use while True in GUI apps

üéì Program Flow Analogy

Think of program as a recipe: Loops = "stir until mixed", Conditionals = "if too thick, add water", Functions = "prepare sauce (see page 5)". Control flow tells computer HOW to execute the recipe!

Python Fundamentals #4

Object-Oriented Programming

Classes, __init__, encapsulation, composition - Production OOP design

Apa itu OOP? (Detailed Explanation)

üìö Definisi Lengkap

Object-Oriented Programming (OOP) adalah paradigma programming yang mengorganisir code sebagai "objects" - bundling data (attributes) dan behavior (methods) together. Ini mirrors how we think about the world: objects memiliki properties dan dapat melakukan actions.

OOP built on 4 core pillars: Encapsulation, Inheritance, Polymorphism, dan Abstraction.

üèõÔ∏è 4 Pilar OOP (Explained)

1. üîí Encapsulation (Data Hiding)

Definisi: Bundling data dan methods dalam satu unit (class), hiding internal state.

Konsep:
  • ‚Ä¢ Data bersifat private (_underscore)
  • ‚Ä¢ Access via public methods
  • ‚Ä¢ Hide implementation details
Contoh:
self._velocity = [0, 0] # Private
def get_speed(self): # Public
2. üå≥ Inheritance (Parent-Child)

Definisi: Class baru inherit properties dari class existing.

Konsep:
  • ‚Ä¢ Child "is-a" Parent
  • ‚Ä¢ Inherit all methods
  • ‚Ä¢ Can override methods
Note:

Prefer Composition over Inheritance!

3. üé≠ Polymorphism (Many Forms)

De finisi: Same method call, different behavior per object type.

for obj in [ball1, ball2]:
  obj.move(0.02) # Works for both!
4. üé® Abstraction (Hide Complexity)

Definisi: Hide complex implementation, show only essential features.

ball.move(0.02) # Simple!
# User tidak tau vector calculations inside

üß© Core Concepts OOP

Class

Blueprint/template untuk create objects. Defines attributes & methods yang akan dimiliki object.

class Ball: # Blueprint
Object (Instance)

Concrete realization dari class. Each instance punya data sendiri tapi share methods.

ball1 = Ball(...) # Instance
Attributes

Data/properties yang dimiliki object. Accessed via self.attribute

self.position, self.velocity
Methods

Functions defined dalam class yang operate on object's data.

def move(self): ...

‚öñÔ∏è Class vs Instance Attributes

Class Attributes
  • ‚Ä¢ Shared by ALL instances
  • ‚Ä¢ Usually UPPERCASE constants
TRAIL_MAX_LENGTH = 30 # Class
Instance Attributes
  • ‚Ä¢ Unique per object
  • ‚Ä¢ Defined in __init__
self.mass = mass # Instance

üéØ Analogi: Pabrik Cookie

üç™
Class = Cookie Mold: Template defines shape & size
üéÇ
Objects = Cookies: Individual cookies dari mold yang sama
üè∑Ô∏è
Class Attributes: "Max baking time = 15min" - semua cookies
üé®
Instance Attributes: Chocolate chip count - beda per cookie
‚ö°
Methods: eat(), crumble() - cookie actions

Teori: Private Methods & Encapsulation

üìö Apa itu Private Methods?

Private methods adalah methods yang dimaksudkan untuk internal use only dalam class. Di Python, ini ditandai dengan _single_underscore prefix (convention, bukan enforcement).

Berbeda dengan bahasa seperti Java/C++, Python tidak enforce private secara strict. Single underscore adalah "gentlemen's agreement" yang berarti: "Please don't touch unless you know what you're doing."

‚úÖ Mengapa Perlu Private Methods?
1. Separation of Concerns

Public methods = "what" (interface), Private methods = "how" (implementation)

2. Flexibility to Change

Internal implementation bisa diubah tanpa break external code

3. Reduced Complexity

User hanya perlu tahu public interface, tidak overwhelmed dengan details

4. Prevent Misuse

Methods yang tidak intended untuk external use jadi clear

üéØ Kapan Buat Private Method?
  • ‚Ä¢ Helper functions yang hanya dipakai internally
  • ‚Ä¢ Setup/initialization methods yang dipanggil dari __init__
  • ‚Ä¢ Implementation details yang mungkin berubah di future
  • ‚Ä¢ Complex logic breakdown - split big public method ke private helpers

Teori: Composition (Has-A Relationship)

üìö Apa itu Composition?

Composition adalah design pattern dimana class "has-a" (memiliki) instance dari class lain sebagai attributes. Ini adalah cara untuk build complex objects dari simpler components.

Contoh Real-World:

‚Ä¢ Car HAS-A Engine (bukan Car IS-A Engine)

‚Ä¢ House HAS-A Kitchen (bukan House IS-A Kitchen)

‚Ä¢ CollisionApp HAS-A Ball (bukan CollisionApp IS-A Ball)

‚úÖ Mengapa Prefer Composition over Inheritance?
1. üîì Flexibility (Fleksibilitas)

Composition: Bisa swap components at runtime. Inheritance: Stuck dengan parent class.

self.ball = FastBall() # Later: self.ball = SlowBall()
2. üß© Loose Coupling (Coupling Rendah)

Composition: Components independent. Inheritance: Tight coupling dengan parent.

3. üîß Easy Testing

Composition: Test components separately. Inheritance: Must test entire hierarchy.

4. ‚ùå Avoid Fragile Base Class Problem

Inheritance: Parent changes break children. Composition: Changes isolated.

    # 4. Get restitution coefficient

    # 5. Calculate impulse magnitude

  5. 5. Update Velocities: Apply impulse to both balls
  6. 6. Separation: Push balls apart if overlapping

Mengapa Code Structure Penting?

üìö Apa itu Code Structure?

Code Structure adalah cara kita mengorganisir dan memformat code agar readable, maintainable, dan consistent. Ini bukan hanya soal "looks pretty", tapi fundamental untuk professional development.

1. Code is Read 10x More Than Written

You'll spend WAY more time reading code than writing it. Good structure = faster comprehension.

2. Team Collaboration

Consistent structure means everyone can understand each other's code instantly.

3. Bug Prevention

Well-structured code makes bugs obvious. Messy code hides bugs in complexity.

4. Professional Standard

Following conventions (like PEP 8) shows you're professional, not amateur.

Why snake_case for variables?

More readable than camelCase in Python. "velocity_x" clearer than "velocityX".

Why 4-space indentation?

2 spaces = hard to see nesting. 8 spaces = too much horizontal space. 4 = perfect balance.

Why import order matters?

Instantly see dependencies. Standard lib (always available) vs third-party (need install) vs local (your code).

Why max 79 characters per line?

Fits on any screen, readable without horizontal scrolling, side-by-side code review.

‚ö° Key Principle: "Code for Humans, Not Machines"

Computers don't care about formatting. Humans do! Good structure makes code self-documenting. If you have to choose between "clever" vs "clear", ALWAYS choose clear.

Clean Code Principles

DRY, KISS, SRP - Write code that lasts

Apa itu Clean Code?

üìö Definisi Clean Code

Clean Code adalah code yang mudah dibaca, mudah dipahami, dan mudah dimodifikasi. Bukan soal "bekerja dengan benar" saja - tapi bekerja dengan cara yang ELEGANT dan MAINTAINABLE.

Seperti kata Uncle Bob Martin: "Clean code adalah code yang ditulis oleh seseorang yang peduli."

‚úÖ Mengapa Clean Code Penting?

1. Mengurangi Technical Debt

Messy code = hutang teknis yang makin besar. Suatu hari harus dibayar dengan refactoring besar-besaran.

2. Mempercepat Development

Clean code lebih cepat dipahami = new features lebih cepat dibuat. Messy code = debugging nightmare.

3. Mengurangi Bugs

Simple, clear code = bugs lebih mudah terlihat. Complex, messy code = bugs tersembunyi di complexity.

4. Team Productivity

Team bisa collaborate lebih efektif. Tidak perlu "tanya ahlinya" untuk setiap perubahan kecil.

üéØ 3 Prinsip Utama Clean Code

1. üîÑ DRY (Don't Repeat Yourself)
üìñ Apa itu DRY?

Setiap piece of knowledge harus punya single, unambiguous representation dalam system. Jangan copy-paste code!

‚úÖ Mengapa DRY Penting?
  • ‚Ä¢ Single Source of Truth: Perubahan di 1 tempat, effect everywhere
  • ‚Ä¢ Maintainability: Fix bug sekali, fixed di semua tempat
  • ‚Ä¢ Consistency: Logic sama dijamin identik
  • ‚Ä¢ Less Code: Fewer lines = fewer bugs
üí° Kapan Extract ke Function?
  • ‚Ä¢ Copy-paste 2+ kali? Extract!
  • ‚Ä¢ Kompleksitas bisa diisolasi? Extract!
  • ‚Ä¢ Bisa diberi nama descriptive? Extract!
  • ‚Ä¢ Might be reused di future? Extract!
2. üíö KISS (Keep It Simple, Stupid)
üìñ Apa itu KISS?

Pilih solusi yang paling simple. Complexity adalah musuh. "Simplicity is the ultimate sophistication" - Leonardo da Vinci.

‚úÖ Mengapa KISS Penting?
  • ‚Ä¢ Easier to Understand: Simple code = cepat dipahami
  • ‚Ä¢ Easier to Debug: Fewer moving parts = easier troubleshooting
  • ‚Ä¢ Easier to Change: Simple structure flexible untuk perubahan
  • ‚Ä¢ Performance: Simple solutions often faster
‚ùå Hindari Over-Engineering
  • ‚Ä¢ Jangan buat abstraction yang tidak perlu
  • ‚Ä¢ Jangan optimize prematurely
  • ‚Ä¢ Jangan gunakan design pattern for the sake of it
  • ‚Ä¢ YAGNI: You Ain't Gonna Need It!

Prinsip: Jika ada 2 solusi yang sama-sama works, pilih yang lebih SIMPLE. Clever code itu bagus untuk ego, tapi CLEAR code bagus untuk team!

3. üéØ SRP (Single Responsibility Principle)
üìñ Apa itu SRP?

Setiap function/class harus punya satu dan hanya satu alasan untuk berubah. One job, one responsibility, one reason to exist.

‚úÖ Mengapa SRP Penting?
  • ‚Ä¢ Easier to Test: One responsibility = simple unit tests
  • ‚Ä¢ Easier to Understand: Function name explains what it does
  • ‚Ä¢ Easier to Reuse: Focused functions reusable di context lain
  • ‚Ä¢ Change Isolation: Perubahan di 1 concern tidak affect others
üí° Cara Detect Violation
  • ‚Ä¢ Function name punya "dan"? Red flag!
  • ‚Ä¢ Function >20 lines? Possibly doing too much
  • ‚Ä¢ Hard to name function? Probably unclear responsibility
  • ‚Ä¢ Banyak parameters? Too many concerns
