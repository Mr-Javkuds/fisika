Object-Oriented Programming

Apa itu OOP? (Detailed Explanation)

üìö Definisi Lengkap

Object-Oriented Programming (OOP) adalah paradigma pemrograman yang mengorganisir kode sebagai "objek" - menggabungkan data (attributes) dan perilaku (methods) menjadi satu. Ini mencerminkan cara kita berpikir tentang dunia: objek memiliki properti dan dapat melakukan aksi.

OOP dibangun atas 4 pilar utama: Encapsulation, Inheritance, Polymorphism, dan Abstraction.

üèõÔ∏è 4 Pilar OOP (Explained)

1. üîí Encapsulation (Menyembunyikan Data)

Apa itu: Seperti tas sekolah - ada barang-barang di dalam (data) tapi orang lain tidak bisa seenaknya ambil. Hanya kamu yang boleh buka tasmu.

Cara Kerja:
  • ‚Ä¢ Data disimpan secara **pribadi** (pakai tanda _garis bawah)
  • ‚Ä¢ Orang lain akses lewat **fungsi khusus** saja
  • ‚Ä¢ Isi dalam "tas" tidak kelihatan dari luar
Contoh:
self._kecepatan = [0, 0] # Pribadi
def ambil_kecepatan(self): # Umum
2. üå≥ Inheritance (Mewarisi Sifat)

Apa itu: Seperti anak mewarisi sifat orang tua. Anak punya kemampuan yang sama dengan orang tua, tapi bisa punya kemampuan tambahan.

Cara Kerja:
  • ‚Ä¢ Anak "adalah" orang tua (IS-A)
  • ‚Ä¢ Dapat semua kemampuan orang tua
  • ‚Ä¢ Bisa ubah kemampuan jadi versi sendiri
Catatan Penting:

Lebih baik pakai Gabungan (Composition) daripada Warisan!

3. üé≠ Polymorphism (Banyak Bentuk)

Apa itu: Seperti remote TV - tombol "ON" yang sama bisa nyalakan berbagai merk TV berbeda. Satu perintah, banyak hasil.

for bola in [bola1, bola2]:
  bola.gerak(0.02) # Fungsi sama, tapi tiap bola punya cara geraknya!
4. üé® Abstraction (Sembunyikan Ribet nya)

Apa itu: Seperti nyetir mobil - kamu cuma perlu tau gas, rem, stir. Tidak perlu tau cara mesin bekerja di dalam.

bola.gerak(0.02) # Gampang!
# Kamu tidak perlu tau perhitungan vektor di dalamnya

üß© Core Concepts OOP

Class

Blueprint/template untuk membuat objek. Mendefinisikan attributes & methods yang akan dimiliki object.

class Ball: # Blueprint
Object (Instance)

Concrete realization dari class. Each instance punya data sendiri tapi share methods.

ball1 = Ball(...) # Instance
Attributes

Data/properties yang dimiliki object. Accessed via self.attribute

self.position, self.velocity
Methods

Functions defined dalam class yang operate on object's data.

def move(self): ...

‚öñÔ∏è Class vs Instance Attributes

Class Attributes
  • ‚Ä¢ Dimiliki oleh SEMUA instance
  • ‚Ä¢ Biasanya konstanta HURUF_BESAR
TRAIL_MAX_LENGTH = 30 # Class
Instance Attributes
  • ‚Ä¢ Unik per object
  • ‚Ä¢ Didefinisikan di __init__
self.mass = mass # Instance

üéØ Analogi: Pabrik Cookie

üç™
Class = Cetakan Kue: Template tentukan bentuk & ukuran
üéÇ
Objects = Kue Cookies: Kue individu dari cetakan yang sama
üè∑Ô∏è
Class Attributes: "Waktu panggang = 15min" - berlaku semua kue
üé®
Instance Attributes: Jumlah choco chip - beda tiap kue
‚ö°
Methods: makan(), hancurkan() - aksi kue

Dalam code: Ball = cetakan, ball_1/ball_2 = kue dengan posisi berbeda!

üèóÔ∏è Class Design

Class Ball - Contoh Lengkap

# ball.py - Class definition
class Ball:
    """
    Representasi objek bola dalam simulasi fisika.
    
    ATRIBUT:
    --------
    canvas : tk.Canvas
        Canvas Tkinter untuk menggambar
    radius_pixels : float
        Jari-jari bola dalam piksel
    mass : float
        Massa bola (kg)
    position : np.array
        Posisi [x, y] dalam meter
    velocity : np.array
        Kecepatan [vx, vy] dalam m/s
    trail_points : deque
        Queue titik-titik jejak bola
    """
    
    def __init__(self, canvas: tk.Canvas, x_pixels: float, y_pixels: float, 
                 radius_pixels: float, color: str, mass: float, 
                 velocity_x: float, velocity_y: float, pixels_to_meters: float):
        """Inisialisasi objek bola."""
        self.canvas = canvas
        self.radius_pixels = radius_pixels
        self.radius_meters = radius_pixels * pixels_to_meters
        self.mass = float(mass)
        self.pixels_to_meters = pixels_to_meters
        
        # Posisi dan kecepatan dalam satuan meter dan m/s
        self.position = np.array([x_pixels * pixels_to_meters, 
                                  y_pixels * pixels_to_meters], dtype=float)
        self.velocity = np.array([velocity_x, velocity_y], dtype=float)

        # Trail (jejak bola)
        self.trail_points = deque(maxlen=TRAIL_MAX_LENGTH)
        self.trail_ids = []

        # Gambar bola di canvas
        self.canvas_id = canvas.create_oval(
            x_pixels - radius_pixels, y_pixels - radius_pixels, 
            x_pixels + radius_pixels, y_pixels + radius_pixels,
            fill=color, outline="black", width=BALL_BORDER_WIDTH
        )

‚úÖ Class vs Instance

  • ‚Ä¢ Class constants: HURUF_BESAR, milik bersama
  • ‚Ä¢ Instance attributes: self.attribute, unik
  • ‚Ä¢ Methods: Fungsi dengan self

üì¶ __init__ Constructor

  • ‚Ä¢ Dipanggil otomatis saat buat objek
  • ‚Ä¢ Inisialisasi atribut instance
  • ‚Ä¢ self = referensi ke instance

Teori: Private Methods & Encapsulation

üìö Apa itu Private Methods?

Private methods adalah methods yang dimaksudkan untuk hanya penggunaan internal dalam class. Di Python, ini ditandai dengan awalan _satu_garis_bawah (konvensi, bukan aturan keras).

Berbeda dengan bahasa seperti Java/C++, Python tidak memiliki private secara ketat. Satu garis bawah adalah "kesepakatan bersama" yang berarti: "Tolong jangan sentuh kecuali kamu tahu apa yang kamu lakukan."

‚úÖ Mengapa Perlu Private Methods?
1. Pemisahan Tugas

Public = "apa" (antarmuka), Private = "bagaimana" (implementasi)

2. Fleksibilitas Perubahan

Implementasi internal bisa ubah tanpa merusak kode luar

3. Mengurangi Kompleksitas

User cukup tahu antarmuka publik, tidak pusing detailnya

4. Cegah Penyalahgunaan

Method yang bukan untuk luar jadi jelas

üéØ Kapan Buat Private Method?
  • ‚Ä¢ Fungsi pembantu yang hanya dipakai internal
  • ‚Ä¢ Method setup/inisialisasi yang dipanggil dari __init__
  • ‚Ä¢ Detail implementasi yang mungkin berubah nanti
  • ‚Ä¢ Pecah logika rumit ke pembantu private
üí° Pattern: Public Interface vs Private Implementation

‚úÖ Public (User-facing):

def start_simulation(self):
  """User calls this"""
  self._run_loop() # Internal

üîí Private (Internal):

def _run_loop(self):
  """Implementation detail"""
  # Complex logic here

üîí Encapsulation (Method Pribadi)

# app.py - Private method (convention)
def _setup_user_interface(self) -> None:
    """Setup seluruh komponen UI."""
    # _ prefix = internal implementation detail
    
# app.py - Private method calling other private methods
def _setup_left_panel(self, parent: ttk.Frame) -> None:
    """Setup panel kiri (canvas visualisasi)."""
    # Implementation details hidden from public API
    
# Public Methods (no underscore)
def start_simulation(self) -> None:
    """Public method - part of class interface."""
    if not self.is_running:
        self.is_running = True
        self._run_loop()  # ‚Üê Calls private method internally

üéØ Umum vs Pribadi (Public vs Private)

  • ‚Ä¢ Public methods: start_simulation() - User calls ini
  • ‚Ä¢ Private methods: _setup_ui() - Pembantu internal, jangan panggil dari luar
  • ‚Ä¢ Konvensi Python: Awalan satu garis bawah _ = "tolong jangan diotak-atik"

Teori: Komposisi (Hubungan Memiliki)

üìö Apa itu Composition?

Komposisi adalah cara menyusun kode dimana class "punya" (memiliki) komponen dari class lain. Ini cara membangun objek keren dari bagian-bagian kecil.

Contoh Nyata:

‚Ä¢ Mobil PUNYA (HAS-A) Mesin (bukan Mobil ADALAH (IS-A) Mesin)

‚Ä¢ Rumah PUNYA (HAS-A) Dapur (bukan Rumah ADALAH (IS-A) Dapur)

‚Ä¢ Aplikasi PUNYA (HAS-A) Bola (bukan Aplikasi ADALAH (IS-A) Bola)

‚úÖ Kenapa Lebih Baik Komposisi daripada Warisan?
1. üîì Fleksibilitas

Komposisi: Bisa ganti komponen saat jalan. Warisan: Terkunci dengan orang tua selamanya.

self.ball = FastBall() # Nanti: self.ball = SlowBall()
2. üß© Ketergantungan Rendah (Loose Coupling)

Komposisi: Komponen mandiri. Warisan: Ketergantungan tinggi dengan orang tua.

3. üîß Mudah Dites

Komposisi: Tes komponen secara terpisah. Warisan: Harus tes seluruh hierarki.

4. ‚ùå Hindari Masalah Warisan Rapuh

Warisan: Orang tua berubah, anak ikut rusak. Komposisi: Aman, perubahan terisolasi.

üéØ Komposisi vs Warisan - Kapan Pakai Apa?
‚úÖ Gunakan Komposisi Saat:
  • ‚Ä¢ Hubungan adalah "punya" (has-a) atau "pakai" (uses-a)
  • ‚Ä¢ Butuh fleksibilitas ganti komponen
  • ‚Ä¢ Ingin ketergantungan rendah
  • ‚Ä¢ Bangun sistem kompleks dari bagian-bagian
  • ‚Ä¢ Pilihan utama kebanyakan kasus!
‚ö†Ô∏è Gunakan Warisan Saat:
  • ‚Ä¢ Hubungan "adalah" (is-a) sejati
  • ‚Ä¢ Polymorphism dibutuhkan
  • ‚Ä¢ Perilaku sama antar tipe mirip
  • ‚Ä¢ Contoh: Anjing IS-A Hewan ‚úì
  • ‚Ä¢ Jarang dalam praktek!
üí° Pola dalam Kode Kita:

CollisionSimulatorApp menggunakan Komposisi secara luas:

  • ‚Ä¢ PUNYA Window Tk (self.root)
  • ‚Ä¢ PUNYA Objek Bola (self.ball_1, self.ball_2)
  • ‚Ä¢ PUNYA Canvas (self.canvas)
  • ‚Ä¢ PUNYA Figure Matplotlib (self.figure)

Kenapa tidak warisan? Karena CollisionApp bukan "is-a" Bola, tapi "punya" (berisi) Bola!

üéì Tips Pro: "Utamakan Komposisi daripada Warisan"

Ini adalah famous design principle dari Gang of Four. Start dengan composition by default, hanya gunakan inheritance jika ada clear "is-a" relationship dan polymorphism requirement.

üß© Composition (Has-A Relationship)

# app.py - CollisionSimulatorApp HAS-A many objects
class CollisionSimulatorApp:
    def __init__(self, root: tk.Tk):
        self.root = root  # ‚Üê Has-A Tk window
        self.style = ttk.Style()  # ‚Üê Has-A Style object
        
        # Has-A Ball objects (created later)
        self.ball_1 = None
        self.ball_2 = None
        
        # Has-A Matplotlib components
        self.figure = None
        self.axes = None
        self.chart_canvas = None
        
        # Has-A Tkinter widgets
        self.canvas = None
        self.combo_mode = None
        self.entry_mass_1 = None
        
# app.py - reset_simulation() creates composed objects
self.ball_1 = Ball(canvas, 50, center_y, ...)  # ‚Üê Composition
self.ball_2 = Ball(canvas, width-50, center_y, ...)

‚úÖ Komposisi (Pakai Ini)

CollisionApp HAS-A Ball
CollisionApp HAS-A Canvas
CollisionApp HAS-A Figure

‚ùå Inheritance (Hindari Kalau Bisa)

CollisionApp IS-A Ball (?)
Too tight coupling
